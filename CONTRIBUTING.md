## 🔄 Рабочий процесс

### Для участников команды

1. **Получи задачу** — возьми Issue из [Project Board](https://github.com/users/RadioPizza/projects/4) или создай новое
2. **Создай ветку** — от `main` с правильным именем (см. ниже)
3. **Работай** — делай коммиты с понятными сообщениями
4. **Пуш** — отправь ветку на GitHub
5. **PR (опционально)** — создай Pull Request если нужны изменения в main
6. **Мерж** — после проверки влей изменения в `main`

### Для внешних контрибьюторов

1. **Форкни** репозиторий
2. **Создай ветку** в своём форке
3. **Сделай изменения**
4. **Создай Pull Request** в основной репозиторий
5. **Дождись review**

## 🌿 Ветки

### Защищённые ветки

| Ветка | Защита | Можно пушить напрямую |
|-------|--------|----------------------|
| `main` | ✅ Защита от удаления, force push, прямых обновлений | ❌ Нет |

> ⚠️ **Важно:** Ветка `main` защищена правилами. Прямой `git push origin main` заблокирован.

### Именование веток

**Формат:** `<тип>/<номер-issue>-<краткое-описание>`

| Тип | Когда использовать |
|-----|-------------------|
| `feat/` | Новая функциональность |
| `fix/` | Исправление бага |
| `refactor/` | Рефакторинг без изменений функциональности |
| `docs/` | Документация |
| `chore/` | Вспомогательные задачи (зависимости, конфиги) |
| `hotfix/` | Срочное исправление |

**Примеры:**

```bash
✅ feat/41-add-status-bar
✅ fix/36-finish-dialog-bug
✅ refactor/61-unify-ui-fonts
✅ docs/24-project-structure
❌ new-feature
❌ oleg-test
❌ fix-bug
```

## ✍️ Коммиты

Мы строго следуем спецификации [Conventional Commits](https://www.conventionalcommits.org/). Это позволяет автоматически генерировать changelog и упрощает навигацию по истории.

### Формат сообщения

```plaintext
<тип>[optional scope]: <описание> #([номер issue])
```

### Основные типы

| Тип | Описание |
|-----|----------|
| `feat` | Новая функциональность |
| `fix` | Исправление ошибки |
| `docs` | Изменения в документации |
| `style` | Изменения, не влияющие на логику кода (форматирование, отступы) |
| `refactor` | Рефакторинг кода без изменения функциональности |
| `perf` | Изменения, улучшающие производительность |
| `test` | Добавление или исправление тестов |
| `build` | Изменения в системе сборки или внешних зависимостях |
| `chore` | Вспомогательные задачи (обновление зависимостей и т.п.) |
| `revert` | Откат коммита |

### Примеры

```bash
git commit -m "fix(ui): remove obsolete safety switch warning Fixes #48"`
git commit -m "feat(settings): add per-camera recording toggle checkboxes #47"
git commit -m "chore: cleanup resource files and remove unused icons"
git commit -m "docs: update start window screenshot"
```

> 💡 Если коммит закрывает issue, обязательно указывай это в теле сообщения или футере, например `Fixes #59` или `Closes #12`.

## 🔀 Pull Request

Когда вы заврешили работу на ветке, выделенной под какую-то задачу, необходимо 
Заголовок должен кратко отражать суть изменений.
Каждый PR должен быть связан с задачей (Issue), которую он решает.
Если что-то сделать не удалось или в коде есть сложные моменты или что-то подобное, то обязательно оставьте "Особые примечания" в тексте PR.

### Шаблон текста PR

Пример хорошей структуры текста PR (рекомендация, можете менять по ситуации):
```
## Связанные задачи
Closes #123

## Что сделано?
- Краткий список изменений

## Зачем это сделано?
- Контекст, причина

## Как тестировать?
- Пошаговая инструкция

## Скриншоты (если применимо)

## Особые замечания
- Если есть сложные места или вопросы к ревьюеру
```

## 📐 Стиль кода

Мы стремимся к читаемому, поддерживаемому и единообразному коду.

### Архитектура

После завершения Issue #39 проект следует паттерну **Model-View-Presenter**:
- **View**: ТОЛЬКО создание и обновление UI, без бизнес-логики
- **Presenter**: бизнес-логика, обработка событий
- **Model**: работа с данными и оборудованием

## 👥 Code Review

Code Review — желательный этап для всех изменений, попадающих в main. Цель: не найти виноватого, а сделать код лучше и поделиться знаниями.
